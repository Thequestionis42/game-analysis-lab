# RPG-like 单产品长线版本策略分析 Skill

## 0. 适用范围

使用本 Skill 分析 **一款** RPG-like 产品的长期版本策略。每次任务只能分析一个产品。

目标不是整理版本日志，而是回答：

> 这是一款怎样的 RPG 产品？它的主线体验由什么构成？主线是否持续稳定供给？进入中后期后，产品依靠哪些内容继续制造玩家目标感？主线、新养成线、旧系统深化、活动、赛季、竞争、社交、生命周期治理和商业化辅助分别承担什么作用？

适用产品包括但不限于：

- 放置 RPG；
- ARPG；
- MMO-lite；
- 卡牌 RPG；
- Roguelike RPG；
- 模拟经营与 RPG 混合产品；
- 其他具有长期数值成长、内容消耗和版本迭代的 RPG-like 产品。

## 1. 非目标与边界

### 1.1 单产品原则

- 每次只分析一个产品。
- 不输出【产品 A】、【产品 B】等多产品章节。
- 不在同一轮任务中做跨产品横向对比。
- 报告末尾可以提炼 RPG-like 产品策略启发，但必须明确这些启发来自当前单一案例，不得写成未经验证的行业普遍规律。

### 1.2 必须保留的辅助轴

商业化分析保留，但仅作为辅助轴：

- 分析新养成线、旧系统深化、阶段版本、活动和赛季是否同步出现抽取、保底、自选、商店、凭证、礼包、充值重置等承载；
- 分析商业化与资源消耗、系统深化和生命周期阶段的关系；
- 不估算收入、付费率、ARPU 或礼包性价比；
- 不评价商业化效果。

### 1.3 禁止输出

不得输出：

- 开发成本估算；
- 人日估算；
- 美术或技术工作量判断；
- 玩家反馈专题；
- 社区舆情专题；
- 付费抱怨、卡关抱怨或玩家接受度分析；
- 具体版本排期建议；
- 直接照搬建议；
- “建议每 X 个月上线某系统”等执行层方案。

可以描述内容复用程度、系统结构变化和生命周期作用，但不得进一步推断研发投入。

## 2. 输入要求

### 2.1 最低输入

任务至少需要：

- 产品名称；
- 版本日志文件、公告链接或抓取结果；
- 截止日期；
- 重点市场与平台；
- 特别关注问题（可选）。

### 2.2 推荐输入模板

参见 `templates/INPUT_TEMPLATE.md`。

### 2.3 输入不足时的处理方式

- 不得编造缺失内容。
- 对无法确认的内容标记为“资料不足”。
- 对已扫描但未找到可靠证据的内容标记为“未发现”。
- “未发现”不得改写为“不存在”。
- 如果版本日志缺少关键区间，必须在正文限制章节和附录资料缺口中列出。

## 3. 核心执行顺序

必须严格按照以下顺序执行：

0. 读取产品理解质量校准范例；
1. 理解产品；
2. 建立前置研究假设；
3. 整理版本历史；
4. 提取严格主线扩容；
5. 提取阶段版本包；
6. 分类养成系统演进；
7. 扫描中后期机制；
8. 分析商业化辅助轴；
9. 归纳生命周期策略；
10. 执行一致性审计；
11. 读取 HTML 报告结构范例；
12. 生成 HTML 报告。

顺序不能反过来。必须先回答：

> 这是一款怎样的游戏？

然后才能回答：

> 它后来更新了什么？为什么这样更新？

## 4. 第一阶段：产品定位与核心玩法结构分析

这一阶段不能只读取版本日志。需要结合公开资料核验系统含义。

### 4.1 来源优先级

按以下顺序优先使用：

1. 官方游戏公告；
2. 官方网站；
3. TapTap 官方动态；
4. App Store、Google Play 或其他应用商店更新日志；
5. 官方社交媒体；
6. 官方宣传视频、实机演示和官方新手指南；
7. 攻略站或资料库，仅用于补充机制解释；
8. 玩家讨论，仅作为辅助线索，不得支撑核心结论。

### 4.2 必须分析的产品结构

#### A. 产品定位

回答：

- 表层包装是什么；
- 核心品类是什么；
- 中层玩法结构是什么；
- 战斗形态是什么；
- 长线驱动力是什么；
- 核心差异点是什么；
- 玩家为什么持续回来。

最终给出一句明确判断：

> 这款产品看起来是什么，以及实际上依靠什么留住玩家。

#### B. 核心循环

分别分析：

| 周期 | 要分析的内容 |
|---|---|
| 分钟级 | 玩家频繁执行的基础操作 |
| 日级 | 日常任务、资源获取和成长行为 |
| 周级 | 周期挑战、多人协作和资源循环 |
| 长周期 | 主线推进、阶段突破、系统解锁和 Build 演进 |

输出循环图或文本链：

```text
基础玩法
→ 获取资源
→ 强化角色或 Build
→ 突破卡点
→ 解锁新阶段
→ 进入新的挑战与成长目标
→ 返回下一轮循环
```

#### C. 主线体验结构

固定分析：

| 字段 | 核心问题 |
|---|---|
| 推进单位 | 小关、章节、地图、区域、世界、难度或转职阶段 |
| 战斗形式 | 自动、半自动、手动、组队或 Boss 节点 |
| 叙事密度 | 重剧情、轻剧情、环境叙事或纯数值推进 |
| 奖励结构 | 经验、金币、装备、挂机收益、资源或系统解锁 |
| 卡点类型 | 数值、Build、操作、时间或社交 |
| 主线作用 | 内容消费、成长坐标、资源产地、系统门槛或剧情载体 |
| 与外围玩法关系 | 是否为副本、养成、活动和商业化提供阶段门槛 |

必须给出一句判断：

> 主线在该产品中，本质上是什么？

#### D. 战斗验证场景

区分：

| 场景 | 常见产品作用 |
|---|---|
| 主线普通关卡 | 基础成长验证 |
| 主线 Boss | 阶段卡点 |
| 日常副本 | 资源供给 |
| 高难副本 | Build 与操作验证 |
| 团本或合作玩法 | 职业分工与社交协作 |
| 排行榜玩法 | 高战竞争出口 |
| 跨服玩法 | 扩大竞争池或社交池 |
| 赛季玩法 | 周期性目标刷新 |

#### E. 初始成长结构

先梳理产品基础养成盘：

| 类型 | 示例 |
|---|---|
| 基础成长 | 等级、装备、技能 |
| 职业成长 | 转职、天赋、技能分支、Build |
| 角色外养成 | 宠物、坐骑、伙伴、藏品、神器 |
| 构筑深化 | 技能风格、词条、装备槽、融合 |
| 收集成长 | 图鉴、收藏、时装属性 |
| 社交成长 | 公会科技、关系等级、协作奖励 |
| 阶段成长 | 转生、觉醒、突破、代际升级 |

#### F. 社交结构

分析：

| 关系类型 | 示例 |
|---|---|
| 弱社交 | 点赞、助战、排行榜 |
| 固定关系链 | 好友、搭子、师徒、伴侣 |
| 组队协作 | 副本、Boss、合作任务 |
| 组织关系 | 公会、旅团、联盟 |
| 竞争关系 | 排行榜、跨服、竞技 |
| 资源关系 | 赠送、交换、交易、拍卖 |
| 内容关系 | UGC、观赛、攻略和 Build 分享 |

#### G. 初始内容供给结构

识别产品主要依靠哪些内容持续更新：

- 主线扩容；
- 阶段扩容；
- 挑战内容；
- 新养成线；
- 旧系统深化；
- 活动；
- 赛季；
- 竞争；
- 社交；
- 生命周期治理；
- 商业化辅助。

#### H. 前置研究假设

在读取完整版本历史前，必须列出待验证问题：

1. 主线是否持续稳定扩容；
2. 主线是否逐渐从内容主体变为成长底盘；
3. 中后期是否更多依赖养成深化；
4. 高难玩法是否持续扩展；
5. 社交、追赶和减负机制是否随生命周期增强；
6. 商业化是否主要绑定新养成线和阶段版本。

## 5. 第二阶段：严格主线扩容分析

### 5.1 关键原则

严格主线扩容与阶段版本包必须拆开统计。

本章只回答：

> 主线是否像月度固定供给一样持续更新？每次通常增加多少关、章节、地图或区域？到后期主线更新是稳定、放慢，还是被其他内容替代？

### 5.2 允许纳入

- 新增主线关卡；
- 新增章节；
- 新地图；
- 新区域；
- 新大陆；
- 新剧情篇章；
- 新主线 Boss；
- 新难度主线；
- 明确标注属于主线的内容。

### 5.3 禁止纳入

- 转职；
- 秘境；
- 绝境；
- 恶龙；
- 团本；
- 排行榜；
- 新养成线；
- 活动；
- 商业化礼包；
- QoL。

### 5.4 日期规则

版本日志可能包含公告日期、生效日期和服务器开放日期。频率统计优先使用：

```text
实际生效日期
→ 若没有，则使用公告发布日期
→ 若仅能识别服务器动态开放，则单独记录，不纳入统一间隔统计
```

每条记录必须保留：

- 公告日期；
- 生效日期；
- 服务器开放规则（若有）；
- 来源；
- 证据等级。

### 5.5 主线扩容明细表

必须输出：

| 生效日期 | 公告日期 | 新增类型 | 新增批量 | 是否可精确量化 | 距上次主线扩容 | 证据等级 | 来源 |
|---|---|---|---:|---|---:|---|---|

### 5.6 主线体量矩阵

每个主线新增节点固定记录：

| 字段 | 说明 |
|---|---|
| `mainline_chapter_count` | 新增章节数 |
| `mainline_stage_count` | 新增小关数 |
| `mainline_map_count` | 新增地图数 |
| `mainline_region_count` | 新增区域、大陆或世界数 |
| `mainline_boss_count` | 新增主线 Boss 数 |
| `mainline_story_density` | 高、中、低或资料不足 |
| `mainline_reuse_level` | 高、中、低或资料不足 |

### 5.7 主线供给稳定性指标

可计算时必须统计：

- 平均更新间隔；
- 中位更新间隔；
- 最短间隔；
- 最长间隔；
- 标准差；
- 有主线扩容的月份占比；
- 连续无主线扩容的最长时间；
- 前期、中期、后期间隔变化；
- 大版本中包含主线扩容的比例；
- 主线扩容与活动、养成、赛季的共现情况。

资料不足时不得强行计算。

### 5.8 主线供给模式判断

根据事实和指标判断最符合实际的一种或多种模式：

- 月度稳定供给型；
- 双月或季度批量型；
- 大版本绑定型；
- 前期高频、后期放缓型；
- 低频底盘型；
- 资料不足。

## 6. 第三阶段：阶段版本包分析

阶段版本包用于分析完整版本策略，不等于严格主线扩容。

### 6.1 纳入范围

- 转职；
- 等级上限；
- 新难度；
- 主线门槛；
- 新秘境、绝境、Boss、团本、梦魇；
- 新养成线；
- 旧系统深化；
- 活动；
- 赛季；
- 排行榜；
- 跨服；
- 社交；
- QoL；
- 回流；
- 追赶；
- 商业化承载。

### 6.2 阶段版本包明细

必须输出：

| 生效日期 | 阶段版本 | 主线扩容 | 挑战内容 | 新养成线 | 旧线深化 | 活动 | 社交 / 竞争 | QoL | 商业化 | 版本作用 |
|---|---|---|---|---|---|---|---|---|---|---|

### 6.3 阶段版本包体量矩阵

固定使用以下字段，不得自由决定哪些模块计数：

| 字段 | 说明 |
|---|---|
| `stage_progression_count` | 转职、等级或难度阶段 |
| `mainline_content_count` | 主线相关新增项 |
| `challenge_content_count` | 秘境、团本、Boss、高难玩法 |
| `new_progression_system_count` | 独立新养成线 |
| `existing_system_deepening_count` | 旧养成线深化 |

### 6.4 主线与阶段版本包关系

必须回答：

- 主线是否仍然是大版本核心内容；
- 主线是否逐渐变为阶段解锁门槛；
- 新养成线是否开始替代主线承担目标感；
- 活动、赛季、排行榜、跨服和社交是否进入主版本结构；
- 主线空窗期由什么内容承接。

## 7. 第四阶段：主线生命周期作用分析

同时使用两个维度：

| 维度 | 含义 |
|---|---|
| 距上线月份 | 上线后第 N 个月 |
| 产品演进阶段 | 初期、中期、后期或长线阶段 |

必须输出：

| 阶段 | 主线更新特点 | 主线承担作用 | 配套内容 | 是否出现替代内容 |
|---|---|---|---|---|

必须回答：

- 前期主线为什么重要；
- 中期主线是否稳定扩容；
- 后期主线是否放慢；
- 哪些内容开始部分替代主线；
- 主线最终变成了什么。

## 8. 第五阶段：养成系统分类与演进

### 8.1 主分类

每个系统只能有一个主分类：

| 主分类 | 判断标准 |
|---|---|
| 独立新养成线 | 有新入口、新资源、新成长闭环 |
| 旧养成线深化 | 在旧系统上增加品质、装备、代际、词条、融合、觉醒、突破、装配或技能分支 |
| 旧系统内容扩充 | 只增加新装备、新宠物、新 Boss、新对象或新皮肤，系统结构未改变 |
| 展示、图鉴、便利功能或 QoL | 主要解决展示、筛选、批量、托管、找回和操作体验 |
| 特殊机制 | 无法合理归入前四类，需要单独解释 |

## 14. RPG-like 产品策略启发

必须讨论：

### 14.1 主线关卡在生命周期中承担什么作用

- 前期：教学、成长反馈、世界观和系统解锁；
- 中期：稳定消耗、数值卡点和 Build 验证；
- 后期：进度坐标、阶段门槛和资源底盘；
- 长线阶段：未必仍是版本卖点，但通常仍是成长骨架。

### 14.2 新养成线适合承担什么作用

- 提供长期目标；
- 增加资源消耗；
- 延展 Build；
- 辅助商业化；
- 在主线放缓后维持成长动力；
- 同时警惕系统膨胀、资源割裂和回流门槛。

### 14.3 旧养成线深化适合承担什么作用

- 提升旧系统上限；
- 降低学习成本；
- 复用既有资源；
- 形成代际追求；
- 通过继承、置换和迁移降低损失感。

### 14.4 活动和限时玩法适合承担什么作用

- 填补主线空窗；
- 制造短周期目标；
- 承接节庆节点；
- 提供资源循环；
- 承载限时商业化；
- 测试新玩法。

### 14.5 排行榜、跨服和难度扩展适合承担什么作用

- 为高战玩家提供验证场；
- 延长存量玩法寿命；
- 增加竞争和展示；
- 扩大人口池；
- 降低持续生产新 PVE 内容的压力。

### 14.6 回流、追赶和减负适合承担什么作用

- 降低回流门槛；
- 缓解系统膨胀；
- 缓解进度差；
- 降低日常疲劳；
- 维持组队池；
- 延长产品生命周期。

必须使用克制措辞：

- 可以写“该产品提供了一种可参考路径”；
- 可以写“从该产品的演进可以观察到”；
- 不得写“行业普遍如此”；
- 不得写“RPG 产品都应该这样做”。

检查：

- 图表是否能由明细复算；
- 体量字段是否统一；
- 主线与副本是否分开；
- 事实、推导和推断是否明确区分；
- 核心结论是否有官方来源；
- 是否将未发现误写为不存在；
- 是否输出了不需要的开发成本估算；
- 是否输出了不需要的玩家反馈分析；
- 是否输出了不需要的具体排期建议；
- 是否意外加入第二个产品的分析章节。

## 18. 最终交付要求

最终必须交付：

1. 一份 HTML 报告；
2. HTML 报告内嵌完整正文、图表和附录；
3. 所有关键事实附来源链接；
4. 所有推断明确标记证据等级；
5. 所有统计数据可通过附录复算；
6. 不输出第二个产品；
7. 不输出开发成本、玩家反馈专题或具体排期建议。

## 19. 默认质量判断

一份合格报告必须满足：

- 先解释产品，再统计版本；
- 严格主线扩容与阶段版本包完全拆开；
- 主线更新频率、批量和阶段变化可以复算；
- 养成系统分类稳定；
- 中后期机制扫描完整；
- 商业化只作为辅助轴；
- 结论区分事实、推导、推断和资料不足；
- 附录足以支撑复核；
- 最终输出能够回答：该产品在生命周期中依靠什么持续制造玩家目标感。
